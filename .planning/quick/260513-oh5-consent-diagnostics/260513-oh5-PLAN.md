# Quick Task: Migration consent UX → Problems-pane diagnostics + Code Actions

## Why

Today's consent UX shows a non-blocking toast per `(entry, case)` group. After fixing the recheck-command wiring (260513-o1k), the user observed that VS Code notifications render plain text only — backticks and `**bold**` show as literal characters. The follow-up `a2faf71` stripped markdown, but the per-action descriptions had to go with it.

Switching to `DiagnosticCollection` + `CodeActionProvider` gets us:
- Persistent surface in the Problems pane (no risk of missing a toast)
- Quick-fix actions that VS Code styles natively
- Free "snooze" — the diagnostic IS the snooze; user acts when ready
- Anchoring on the actual `settings.json` line where the legacy key lives

## Design (settled in conversation — see Q1-Q4 chat 2026-05-13)

| Choice | Pick |
|---|---|
| Surface | **B** — diagnostics + single "N migrations available — see Problems pane" summary toast |
| Anchor | **C** — per-scope file URI, JSONC-aware range pointing at the legacy key |
| Severity | `Warning` |
| Snooze | Free — persistent diagnostic; existing `Don't migrate`/`Keep both` clear it |

## Implementation

### 1. New module `src/migrations/diagnostics.ts`

- `MIGRATION_DIAG_SOURCE = 'gs-behave-bdd'` constant.
- Lazy singleton `getDiagnosticCollection()` returning a `vscode.languages.createDiagnosticCollection('gs-behave-bdd.migrations')`. Module also exports `disposeDiagnosticCollection()` for test cleanup.
- `resolveAnchor(scope, wkspUri)` → `{ uri, key }` where `key` is the JSON property path to look for. Per-scope resolution:
  - `Global` → user settings.json: prefer `vscode.env.appRoot` introspection? No — simpler: cross-platform fallback via `process.platform` + `os.homedir()`:
    - win32 → `%APPDATA%/Code/User/settings.json`
    - darwin → `~/Library/Application Support/Code/User/settings.json`
    - linux → `~/.config/Code/User/settings.json`
    - (VS Code Insiders ignored — out of scope; fall back to `[0,0]` if file missing.)
  - `Workspace` → `vscode.workspace.workspaceFile` (the `.code-workspace`).
  - `WorkspaceFolder` → `<wkspUri>/.vscode/settings.json`.
- `computeRange(uri, propertyPath)` async helper:
  - Read file via `vscode.workspace.fs.readFile`.
  - Use `jsonc-parser`'s `parseTree()` to build the AST, then `findNodeAtLocation(root, propertyPath)` to locate the property's key node.
  - Convert the node's `offset`/`length` to `vscode.Position`/`Range` using a line/column scan of the buffer.
  - Fallback: `new vscode.Range(0, 0, 0, 0)` on any error (file missing, parse error, key absent, .code-workspace path mismatch for Workspace scope which nests under `"settings"`).
- For `Workspace` scope (when anchor is `.code-workspace`), the key path is `['settings', '<namespace>.<key>']`.
  For `Global` and `WorkspaceFolder`, it's `['<namespace>.<key>']`.
  Note: VS Code stores legacy `behave-vsc.justMyCode` either as a single flat property `"behave-vsc.justMyCode"` OR under a nested object `"behave-vsc": { "justMyCode": ... }`. Try flat first (the common case in VS Code's settings.json), then nested as a fallback.
- `publishConsentDiagnostics(wkspUri, hits)`:
  - Group hits by anchor URI.
  - For each URI, build an array of `vscode.Diagnostic` (severity = `Warning`, `source = MIGRATION_DIAG_SOURCE`, `code = ${entry.id}::${case}::${scope}`, `message` = friendly one-liner like `"behave-vsc.<key> is set in <Scope>. Migrate to gs-behave-bdd.<key>?"` or for case 3 `"Both behave-vsc.<key> and gs-behave-bdd.<key> are set in <Scope>."`).
  - **Replace** that URI's existing entries via `collection.set(uri, diagnostics)` — fresh state each activation.
- `clearDiagnosticsForEntryAtScope(entry, scope, wkspUri)`:
  - Iterate all URIs in the collection; filter their diagnostics removing any whose `code` matches `${entry.id}::*::${scope}` (any case for that entry+scope). Re-set the filtered list per URI.

### 2. New module `src/migrations/codeActions.ts`

- Command id: `gs-behave-bdd.migration.action`.
- `MigrationCodeActionProvider implements vscode.CodeActionProvider`:
  - `provideCodeActions(document, range, context)`:
    - For each `diagnostic` in `context.diagnostics` where `diagnostic.source === MIGRATION_DIAG_SOURCE`:
      - Parse `code` (`<entryId>::<case>::<scope>`).
      - Build either case-2 (3 actions) or case-3 (4 actions) `vscode.CodeAction[]`:
        - `kind = vscode.CodeActionKind.QuickFix`
        - `title` mirrors today's button labels (`'Migrate & delete'`, etc.).
        - `diagnostics = [diagnostic]` (so VS Code links the action to the diagnostic).
        - `command = { command: 'gs-behave-bdd.migration.action', title, arguments: [{ entryId, case, scope, action, wkspUri: document.uri.toString() /* NB: this is the anchor file uri, NOT the workspace uri — see dispatch */ }] }`
    - Returns the flat list.
- The wkspUri for **action dispatch** must be the workspace folder uri (not the anchor file). We resolve it inside the dispatch command from the scope: Global/Workspace use any workspace folder (or undefined); WorkspaceFolder finds the matching folder via `vscode.workspace.getWorkspaceFolder(document.uri)` — falling back to the first folder when the anchor file isn't inside a known folder (Global anchors at user settings.json).
- Dispatcher `dispatchMigrationAction({ entryId, case, scope, action })` exported from `codeActions.ts`:
  - Look up `MigrationEntry` by `entryId` from `MIGRATION_REGISTRY`.
  - Resolve `wkspUri`: prefer the first workspace folder; fall back to a no-op log if none.
  - Switch on `action` and invoke the existing handler from `consent.ts` (which we'll expose).
  - On success, call `clearDiagnosticsForEntryAtScope(entry, scope, wkspUri)`.

### 3. Refactor `src/migrations/consent.ts`

- Export the seven action handlers (`runMigrateAndDelete`, `runMigrateAndKeep`, `runDontMigrate`, `runOverwriteAndDelete`, `runOverwriteAndKeep`, `runKeepCanonicalAndDeleteLegacy`, `runKeepBoth`) so `codeActions.ts` can route to them.
- Rewrite `runConsentFlow` body:
  - Compute groups exactly as today (preserve grouping logic — tests assert it).
  - Apply silent migrationMode dispatch for case-2 groups when mode !== 'prompt' (no diagnostic, no toast). Unchanged.
  - For every group that would otherwise prompt (case 2 + mode==='prompt', OR any case 3): collect into a "to-publish" list; do NOT call `showInformationMessage` per group.
  - After processing groups:
    - If to-publish list non-empty:
      - `publishConsentDiagnostics(wkspUri, hitsForThisWkspUri)` — uses the original `hits` filtered to the prompt-bound groups.
      - Show ONE summary toast: `${N} legacy behave-vsc setting${N>1?'s':''} need attention. Open the Problems pane to choose what to do.`
        - This toast has no buttons. Dismissal is fine — diagnostics persist.
    - reloadSettings call: keep the existing one (it ran when hits.length > 0).
- `formatCase2Message` / `formatCase3Message`: keep them but mark unused for prompt path. They're still used in diagnostic messages so refactor them to be the diagnostic message builders (plain text, no markdown — already done in `a2faf71`).
  Decision: rename to `formatCase2DiagnosticMessage` / `formatCase3DiagnosticMessage` to make the role explicit; rebuild prompt-test expectations against the new names.
  → On second thought, the existing tests for `formatCase2Message` were never importing them; only the inline message-content assertions matter. We keep the exported names for minimal churn.

## Risk register

- **JSONC range fallback** — if `jsonc-parser` can't find the key (legacy uses nested object syntax we didn't anticipate), the diagnostic anchors at line 0. UX degrades gracefully; no functional bug.
- **User settings.json path on non-default VS Code installs** (portable mode, Insiders, custom `--user-data-dir`) — we use the default path; mismatches fall back to `[0,0]` on the default path. Document the limitation in the SUMMARY.
- **Code Action discoverability** — the user has to open the Problems pane or the relevant settings.json for the quick-fix bulb to appear. The summary toast mentions Problems pane to address this.
- **Test churn in `consent.test.ts`** — meaningful, but the new tests cover the same dispatch surface via `dispatchMigrationAction`. Net coverage should be flat or up.

## Non-goals

- No change to `migrationMode` setting or its silent paths.
- No change to `MIGRATION_REGISTRY` or evaluator logic.
- No change to in-namespace migrations (`featuresPath-self` etc.) — they don't go through `runConsentFlow`.

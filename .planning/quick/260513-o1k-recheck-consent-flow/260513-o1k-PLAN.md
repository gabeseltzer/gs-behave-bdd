# Quick Task: Wire runConsentFlow into recheckMigrations command

## Problem

`gs-behave-bdd.recheckMigrations` clears `completedMigrations` and re-runs the evaluator,
but never invokes `runConsentFlow`. So case-2 / case-3 hits are classified and dropped —
no prompt ever fires. User noticed: set `behave-vsc.justMyCode: false` at user level,
ran "recheck migrations", got nothing.

Root cause: `recheckCommand.ts` passes an optional `hooks?` parameter through to the
evaluator, but the command registration at `src/extension.ts:428` calls
`recheckMigrationsCommandHandler()` with no hooks. Production has no hits-collector,
no `runConsentFlow` call.

## Fix (Option A — orchestration owned by recheckCommand)

In `src/migrations/recheckCommand.ts`:
- Drop the `hooks?` parameter from `recheckMigrationsCommandHandler`.
- Inside the per-folder loop, mirror the activation flow at `extension.ts:341-356`:
  - Build a local `hits: ConsentHit[]`.
  - Pass an `onCaseHit` hook that pushes case-2 / case-3 hits.
  - Read `migrationMode` via `readMigrationMode(folder.uri)`.
  - `await runConsentFlow(folder.uri, hits, mode)` (sequential await — user just
    triggered the command, so we want the prompt to be visible synchronously).

No change at the `extension.ts:428` registration site.

## Test

Add `test/unit/migrations/recheckCommand.test.ts`:
- Stub the quick-pick to choose Global scope.
- Stub `getConfiguration` so that, after the `completedMigrations` clear, a legacy
  setting (`behave-vsc.justMyCode: false`) is set at Global while canonical is unset.
- Stub `vscode.window.showInformationMessage` and assert it was called once with the
  case-2 button set — proving the consent flow ran.

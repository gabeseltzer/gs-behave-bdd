## Regressions Fixed During UAT

The full integration run surfaced three pre-existing breakages that were
hiding behind the multiroot-mutex flake (integration tests were rarely run
locally). All fixed within UAT scope; details in commits on this branch.

1. **`project-switch suite > switch to beta and verify tree rebuilds`** —
   Phase 19 / CLEANUP-02 added `clearActiveProjectCache()` to
   `configurationChangedHandler` but never repopulated the project list,
   so multi-project workspaces with no root-level config lost their
   `setActiveProject` choice on any `forceFullRefresh`. Fix: re-scan and
   `rebuildProjectList` for every workspace when `needsRescan` (mirrors
   the activation flow). `rebuildProjectList` calls `restoreOrAutoSelectActive`,
   which restores the persisted selection if it's still in the new list —
   so `discoveryDepth=0` correctly drops subdir selections (monorepo-scan
   suite) while explicit `setActiveProject(beta)` survives (project-switch
   suite). [commits: 7b34307, c1e8a81]
2. **`migrations suite > post-activation settings.json: legacy keys removed,
   canonical keys written`** (plus 2 cascading tests) — Phase 21's consent
   UX made featuresPath migrations case-2 prompt-by-default. The
   `migration-stale` fixture predates the consent flow and had no
   `migrationMode`, so the default-dismiss stub blocked all migrations.
   Fix: add `gs-behave-bdd.migrationMode: "migrate-and-delete"` to the
   fixture so case-2 takes the silent path. ALSO surfaced a related cache
   regression: Phase 21's fire-and-forget `runConsentFlow` broke the D-18
   contract that `WorkspaceSettings` cache reflects post-migration state.
   Fix: `runConsentFlow` now calls `config.reloadSettings(wkspUri)` once
   at the end (only when there was at least one consent hit). Unit-test
   stub updated to keep the consent.ts test suite green.
3. **`config-only`, `pyproject-config`, `malformed-config` suites — `nodeCount`
   mismatch (silent)** — these fixtures were committed Apr 16; the
   path-group TestItem was added Apr 20 (cf3c381). The assertion error was
   caught-and-logged but not re-thrown, so mocha saw the suites passing
   while console.error spammed the log. Fix: bump expected `nodeCount`
   from 4 to 5 in all three `expectedResults.ts` files. [commit: a09dfea]

After all three fixes: full integration run finishes with `test run
complete`, exit code 0, all 20 suites green.

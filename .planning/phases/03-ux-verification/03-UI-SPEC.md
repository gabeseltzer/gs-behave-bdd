# Phase 3 — UI Design Contract

> Visual and interaction contract for Phase 3: UX & Verification.
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.
>
> **Platform note:** This is a VS Code extension. There is no web UI, no React/Next.js/Vite,
> no CSS framework, and no shadcn. All "UI" surfaces are VS Code extension APIs:
> output channel text, notification strings, status bar items, Problems panel diagnostics,
> and `package.json` setting descriptions. Design tokens (spacing, web typography, hex colors)
> do not apply. This contract covers copywriting, interaction behavior, and VS Code surface contracts.

### Setting Descriptions (UX-05)

Replace the current `markdownDescription` values in `package.json` with:

**`gs-behave-bdd.projectPath`:**

```
*workspace-relative* path to the root of your behave project (working directory for behave).
**Override only:** Leave blank to use auto-discovery from `behave.ini`, `.behaverc`, `setup.cfg`,
`tox.ini`, or `pyproject.toml`. Set this only if auto-discovery does not detect the correct
project root. Example: `myproject/backend`.
```

**`gs-behave-bdd.featuresPath`:**

```
*project-relative* path to the features subfolder.
**Override only:** Leave blank to use the path from your behave config file, or `features/` if
no config file is found. Set this only if auto-discovery resolves the wrong features directory.
This path is relative to `projectPath` (or workspace root if `projectPath` is not set).
Example: `my_behave_tests`.
```

Source: CONTEXT.md D-08; existing values in `package.json` lines 35 and 41.

## Interaction Behavior Contract

### Status Bar Click Action (D-07 — Claude's Discretion)

**Decision:** Click opens the workspace output channel for the primary workspace.

**Rationale:** The status bar item already surfaces discovery metadata in `detail` (hover). The most
useful next action for a developer seeing an unexpected discovery source is to read the full log.
"Open Settings" is already available as a second button in the warning notification when things go
wrong; the status bar click should be a distinct, always-available action. Opening the output channel
is consistent with how VS Code language extensions handle status-item clicks (see: Pylance, ESLint).

**Implementation:** `statusItem.command = { title: "Open Behave BDD Output", command: "gs-behave-bdd.openOutputChannel" }` or use the existing output channel show API.

Source: CONTEXT.md D-07 (Claude's Discretion).

### Status Bar Visibility Rules (D-07 — Claude's Discretion)

**Decision:** Always visible when at least one workspace folder has features (i.e. `getUrisOfWkspFoldersWithFeatures().length > 0`).

**Rationale:** The status bar item is scoped to `{ language: 'gherkin' }` — it only appears in `.feature`
file editors regardless. Additional hide/show logic based on discovery source would add complexity
without user benefit. "Always visible in feature files" is the simplest mental model.

Source: CONTEXT.md D-07 (Claude's Discretion).

### Duplicate Notification Guard

One malformed-config notification per config file URI per session. Track fired notifications in a
module-level `Set<string>` (keyed by `configFileUri.fsPath`). Clear the Set only on
`forceFullRefresh` (workspace folder changes). This prevents repeated popups on every
`configurationChangedHandler` invocation.

Source: RESEARCH.md Pitfall 2.

### Diagnostic Lifecycle

| Event | Action |
|-------|--------|
| Discovery finds config parse error | `setConfigParseErrorDiagnostic(configFileUri, errorMessage)` |
| Discovery succeeds (no error) for a URI that previously had an error | `clearConfigParseErrorDiagnostic(configFileUri)` |
| `configurationChangedHandler` fires | Clear all config diagnostics for affected workspace before re-running discovery |
| Extension deactivates | `config.diagnostics.dispose()` handles cleanup (existing pattern) |

Source: RESEARCH.md Pitfall 3.

1. Add `setProjectSwitchInProgress` to the existing import from `'./common'`.

2. Convert the `onDidAccept` callback to `async` and add rebuild logic after the existing `updateProjectStatusBar` call. The full replacement for the `if (!wasActive)` block:

```typescript
if (!wasActive) {
  setActiveProject(targetWkspUri, selected.entry);
  // D-09: Log switch to output channel
  const configType = selected.entry.configFileUri.path.split('/').pop() ?? 'config';
  const displayLabel = selected.entry.label === '.' ? '(root)' : selected.entry.label;
  config.logger.logInfo(`Active project switched to: ${displayLabel} (${configType})`, targetWkspUri);
  updateProjectStatusBar(targetWkspUri);

  // Phase 14: Trigger full rebuild after project switch (INT-01, INT-02)
  // D-04: Block test runs during rebuild via projectSwitchInProgress flag
  // D-02: Show withProgress notification during rebuild
  // D-03: No confirmation notification after — tree update is sufficient
  setProjectSwitchInProgress(true);
  vscode.window.withProgress(
    { location: vscode.ProgressLocation.Notification, title: `Switching to project: ${displayLabel}...` },
    async () => {
      try {
        await configurationChangedHandler(undefined, undefined, true);
      } finally {
        setProjectSwitchInProgress(false);
      }
    }
  );
}
```

Key design points:
- `withProgress` is fire-and-forget (not awaited) — the quick-pick is already hidden and the user sees the notification. Awaiting would require the entire `onDidAccept` callback to block, which is unnecessary.
- `configurationChangedHandler(undefined, undefined, true)` triggers the proven full rebuild path: clear scan cache → re-scan → reload settings → recreate watchers → reparse all workspaces → update discovery UX. This is the same call used by integration tests.
- The `finally` block ensures `setProjectSwitchInProgress(false)` runs even if rebuild throws.
- D-01 (loading spinner in Test Explorer): `parser.clearTestItemsAndParseFilesForAllWorkspaces()` inside `configurationChangedHandler` triggers the VS Code test controller's resolving state automatically.

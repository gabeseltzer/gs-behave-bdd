Create `src/discovery/projectList.ts` with the following:

**Types:**
```typescript
export interface ProjectEntry {
  configFileUri: vscode.Uri;
  dirUri: vscode.Uri;
  depth: number;
  configPriority: number;
  label: string;  // workspace-relative dir path, e.g. "backend" or "apps/api"
}
```

**Module state:**
- `projectListCache`: `Map<string, ProjectEntry[]>` keyed by `uriId(wkspUri)` — stores all discovered projects per workspace in scanner order (depth ASC, configPriority ASC)
- `activeProjectCache`: `Map<string, ProjectEntry | undefined>` keyed by `uriId(wkspUri)` — stores the active project per workspace
- `workspaceStateRef`: `vscode.Memento | undefined` — set once during init, used for persistence

**Persistence key format:** `"gs-behave-bdd.activeProject.<uriId(wkspUri)>"` → stores `{ configFilePath: string }` (the `configFileUri.toString()` of the active project). Using a string path (not the full ProjectEntry) keeps the persisted data minimal and avoids serialization issues with Uri objects.

**Exported functions:**

1. `initProjectListPersistence(workspaceState: vscode.Memento): void` — stores the Memento reference. Called once from `activate()`.

2. `rebuildProjectList(wkspUri: vscode.Uri, scanResult: ScanResult, rootConfigEntry?: ScanResultEntry): ProjectEntry[]` — Converts `ScanResult.primary` + `ScanResult.alsoFound` into `ProjectEntry[]` ordered by scanner order. If `rootConfigEntry` is provided (from `findBehaveConfig` at workspace root), insert it at position 0 (depth 0). Labels are workspace-relative paths via `vscode.workspace.asRelativePath(entry.dirUri, false)`. Stores in `projectListCache`. Then calls `restoreOrAutoSelectActive(wkspUri)` to set the active project. Returns the list.

3. `restoreOrAutoSelectActive(wkspUri: vscode.Uri): ProjectEntry | undefined` — Internal (not exported). Reads persisted active from `workspaceState` using the persistence key. If persisted config URI matches an entry in the current list, sets that as active. Otherwise, auto-selects the first entry in scanner order (per D-06: shallowest depth, highest config priority). Persists the selection. Returns the active entry.

4. `getProjectList(wkspUri: vscode.Uri): ProjectEntry[]` — Returns `projectListCache.get(uriId(wkspUri)) ?? []`.

5. `getActiveProject(wkspUri: vscode.Uri): ProjectEntry | undefined` — Returns `activeProjectCache.get(uriId(wkspUri))`.

6. `setActiveProject(wkspUri: vscode.Uri, entry: ProjectEntry): void` — Sets the active project in `activeProjectCache` and persists to `workspaceState`. Validates that `entry` exists in the current project list (by matching `configFileUri` via `urisMatch`). If not found, throws.

7. `removeProjectByConfigUri(wkspUri: vscode.Uri, configUri: vscode.Uri): { removed: boolean; newActive?: ProjectEntry }` — Removes the project with matching `configFileUri` from the list. If the removed project was the active one: auto-select the next project in scanner order (D-01). If no projects remain: clear active (D-02). Returns `{ removed, newActive }` so the caller can decide whether to notify.

8. `addProjectFromScanEntry(wkspUri: vscode.Uri, entry: ScanResultEntry): ProjectEntry` — Adds a new project entry to the list, maintaining scanner order (insert at correct position by depth then configPriority). Does NOT change the active project (D-07). Returns the new ProjectEntry.

9. `clearProjectList(wkspUri: vscode.Uri): void` — Clears project list and active for this workspace. Used for cleanup.

10. `isManualProjectPathMode(wkspUri: vscode.Uri): boolean` — Checks if `projectPath` is explicitly set via `hasExplicitSetting`. When true, the project list should not be used — the workspace is in single-project manual mode (DISC-04). Read the workspace configuration inline using `vscode.workspace.getConfiguration("gs-behave-bdd", wkspUri)` and the legacy config.

**Implementation notes:**
- Import `uriId`, `urisMatch`, `hasExplicitSetting` from `../common`
- Import `ScanResultEntry`, `ScanResult` from `./configScanner`
- Import `diagLog` from `../logger` for diagnostic logging
- Use `diagLog()` for state changes (active project changed, project added/removed)
- `workspaceState.update()` returns a Thenable — fire-and-forget (no await needed, VS Code handles persistence)
- The label for root-level configs (depth 0) will be `"."` or the workspace folder name — use `vscode.workspace.asRelativePath(dirUri, false)`, which returns `"."` for workspace root

  get<T>(key: string): T | undefined;
  get<T>(key: string, defaultValue: T): T;

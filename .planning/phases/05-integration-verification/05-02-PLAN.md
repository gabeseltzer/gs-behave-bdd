<objective>
Create the shared predicate-polling helper that Phase 5 watcher tests use to wait for the Test Explorer to reach an expected state after a fixture mutation. The helper replaces brittle `setTimeout(N)` waits (per D-11) with a predicate-poll loop (interval 100ms, timeout 5000ms per D-12) and produces a descriptive failure message including the last-seen state when the timeout expires. It is generic so callers can return anything — a matching `vscode.TestItem`, a `DiscoveryEntry`, a tuple — and receive that typed value back on success.

Purpose: Gate the watcher integration tests on observable state rather than wall-clock sleeps. A 5000ms timeout comfortably covers the 500ms configWatcher debounce + parse time + CI slack, and the predicate shape lets tests express "source is 'convention' AND scenario X is visible" as a single wait.

Output: `test/integration/suite-shared/waitForTestTree.ts` exporting `waitForTestTree<T>()` and `WaitOptions`.

```typescript
export interface WaitOptions {
  intervalMs: number;
  timeoutMs: number;
}

export async function waitForTestTree<T>(
  predicate: () => T | undefined,
  options: WaitOptions
): Promise<T>;
```

Semantics:
- Calls `predicate()` immediately, then every `intervalMs` until it returns a truthy (non-`undefined`, non-`null`, non-`false`, non-`0`, non-empty) value or `timeoutMs` elapses.
- On success: resolves with the predicate return value (type `T`).
- On timeout: throws an `Error` whose message includes the numeric `timeoutMs` and a JSON-serialized representation of the last-seen predicate return value (for debugging flaky tests on CI).
- Does NOT swallow exceptions thrown by the predicate — they propagate to the caller.

    ```typescript
    // Predicate-polling helper for integration tests.
    // Waits until `predicate()` returns a truthy value or the timeout elapses.
    // Resolves with the matching value; throws with the last-seen value on timeout.

    export interface WaitOptions {
    	intervalMs: number;
    	timeoutMs: number;
    }

    export async function waitForTestTree<T>(
    	predicate: () => T | undefined,
    	options: WaitOptions
    ): Promise<T> {
    	const start = Date.now();
    	let lastSeen: T | undefined;
    	while (Date.now() - start < options.timeoutMs) {
    		lastSeen = predicate();
    		if (lastSeen !== undefined && lastSeen !== null && lastSeen !== false) {
    			return lastSeen;
    		}
    		await new Promise<void>(t => setTimeout(t, options.intervalMs));
    	}
    	let lastSeenRepr: string;
    	try {
    		lastSeenRepr = JSON.stringify(lastSeen);
    	} catch {
    		lastSeenRepr = String(lastSeen);
    	}
    	throw new Error(
    		`waitForTestTree: predicate did not match within ${options.timeoutMs}ms. ` +
    		`Last seen: ${lastSeenRepr}`
    	);
    }
    ```

    Notes:
    - The `lastSeen !== undefined && lastSeen !== null && lastSeen !== false` check is intentionally stricter than the PATTERNS.md stub (`!== undefined` only) — callers often return `boolean | undefined` for "state reached?" predicates, and returning `false` must NOT satisfy the wait. Also guards against `null` for defensive compatibility.
    - The try/catch around `JSON.stringify` handles the rare case where the last-seen value contains a cycle (e.g. a TestItem with parent-child pointers) — the fallback is `String(lastSeen)`.
    - No imports needed — `setTimeout` and `Date` are globals.
    - Do NOT import `vscode` — this helper must remain usable in any integration suite without extension-host coupling at the helper level (callers get vscode directly).

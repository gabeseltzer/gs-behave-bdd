**Strategy:** Since the command and status bar are defined inside `activate()` as closures, the tests cannot import them directly. Instead, test the **logic extracted into testable units**. The executor should:

1. If Plan 01 extracted `updateProjectStatusBar` logic into a module-level or exported helper, test that directly.

**Test cases to cover:**

a) **Quick-pick item building:**
- Given 3 projects where 2nd is active → items array has 3 entries, 2nd has `"✓ active"` in description
- Given a project with `label: "."` → quick-pick item label is `"(root)"` (D-03)
- Each item has `description` with config file type (e.g. "behave.ini"), `detail` with full fsPath (D-02)
- Each item has `buttons` array with open-config button

b) **Status bar visibility:**
- 0 or 1 projects → status bar hidden
- 2+ projects with active → status bar shows `"Behave: <label>"`
- isManualProjectPathMode returns true → status bar hidden regardless of project count
- Root project (label ".") → status bar shows `"Behave: (root)"` (D-03)
- Tooltip contains project count and "click to switch" (D-07)

c) **Project switch action:**
- Selecting a non-active project → `setActiveProject` called with correct entry
- Selecting the already-active project → `setActiveProject` NOT called (no-op)

d) **Edge cases:**
- No workspace folders → warning message shown
- Empty project list → warning message shown
- isManualProjectPathMode → informational message about removing projectPath

Follow the existing test file patterns:
- Use `describe/it` blocks with Mocha
- Use sinon stubs for `vscode.window.*`, `vscode.commands.*`
- Restore stubs in `afterEach`
- Import from `../../src/discovery/projectList` for type references

If the implementation in Plan 01 keeps all logic in closures (not extracted), write the tests to exercise the registered command via `vscode.commands.executeCommand('gs-behave-bdd.selectProject')` if the mock supports it, OR extract the testable logic into a helper module during this task (create `src/discovery/selectProjectHelpers.ts` with pure functions for item building and visibility logic, then test those).

# v1.4.0 Milestone Review (Phases 15–18)

## Executive Summary

The milestone delivers two focused, well-scoped changes (notification suppression infra + `featuresPath → featuresPaths[]` deprecation) and the migration primitive (`migrateScopedSetting`) is genuinely a clean abstraction. That said, three things should be addressed before this is treated as shipped:

1. **Migration loop is needlessly serial across workspaces** (`extension.ts:323-335`) — it `await`s two migrations per workspace folder before moving to the next one, violating the activation-fast contract documented at line 199 ("THIS MUST RETURN FAST"). One slow `cfg.update` (read-only fs, sync extension host) stalls every other workspace's discovery UX. Run the per-workspace migrations concurrently (`Promise.all` across folders) or fire-and-forget like the existing notification pattern.
2. **`config.reloadSettings` not awaited (Pitfall 8) is wrong-headed in this context.** The comment at `extension.ts:328` says "sync void — Pitfall 8: do NOT await", and the function signature *is* synchronous, but the migration loop is `await`ing the migrations *expressly* so settings cache will be fresh when the post-loop notification fires. If `reloadSettings` is sync that's fine, but every call site inside the same `try` reads `config.workspaceSettings[uri.path]` indirectly through `isSuppressed` — and `workspaceSettings` getter (configuration.ts:77-97) re-creates `WorkspaceSettings` lazily, which in turn calls `getActualWorkspaceSetting`. That means the migration write you just performed at WorkspaceFolder scope races against VS Code's own config-change event delivery; `reloadSettings` snapshots the *current* `getConfiguration()` result, which on some VS Code builds has not yet observed the just-written value. This is the exact reason `cfg.update()` returns a Thenable — to wait for propagation. The migration helpers do `await cfg.update(...)`, so the snapshot read in `reloadSettings` should be consistent on the next tick — but only because of that prior await, not because of any guarantee from VS Code. Worth a comment that pins down *why* the ordering is safe.
3. **Migration notification text is publisher-coupled and not localizable.** `extension.ts:349` hardcodes `@ext:gabeseltzer.gs-behave-bdd`. If the publisher ever changes (rebrand, fork, marketplace transfer), this command silently opens an empty Settings page. There's no test coverage for this exact string. The notification body itself uses backticks for code formatting — `vscode.window.showInformationMessage` does NOT render markdown, so users see the literal `` ` `` characters. Both are user-visible regressions from a normally-polished extension.

The remainder of the findings are around (a) duplication that survived the 16-02 extraction, (b) a real disposable/listener leak risk in the multi-workspace notification path, and (c) test-mock surface that has drifted from production behavior in a way that could mask future regressions.

## BLOCKER findings

### B-01 — Migration notification body is not markdown-rendered; users see literal backticks
**File:** `src/extension.ts:343`
**Issue:** `showInformationMessage` displays plain text. The string `` "Migrated `featuresPath` → `featuresPaths`. The deprecated `featuresPath` setting has been moved to the new `featuresPaths` array." `` will render with literal `` ` `` characters around `featuresPath` and `featuresPaths`. Compare with the multi-config notification at `extension.ts:168` which deliberately avoids backticks. This is a user-facing polish bug on a one-shot migration that every existing user will see exactly once.
**Fix:** Drop the backticks. Suggested copy:
```ts
"Behave BDD: migrated your 'featuresPath' setting to the new 'featuresPaths' array. 'featuresPath' is deprecated and will be removed in a future release."
```

### B-02 — `@ext:gabeseltzer.gs-behave-bdd` hard-coded; brittle to publisher changes
**File:** `src/extension.ts:349`
**Issue:** The command argument hard-codes the publisher ID. If `package.json` `publisher` ever changes, this notification's "Open Settings" button becomes a no-op (opens empty Settings UI) with zero error, zero test, zero log. The 16-01-SUMMARY note acknowledges this is brittle ("Publisher confirmed in 16-01-SUMMARY"). The other notification opens settings via `'gs-behave-bdd'` (a *search query*, line 139) which is publisher-independent and works equivalently.
**Fix:** Replace with the same form used by the existing multi-config notification:
```ts
vscode.commands.executeCommand("workbench.action.openSettings", "gs-behave-bdd.featuresPaths");
```
That deep-links to the specific setting and survives publisher renames.

### B-03 — Per-workspace migration loop is serial; violates activate()-fast contract
**File:** `src/extension.ts:323-335`
**Issue:** The activate() banner at line 199 explicitly says "THIS MUST RETURN FAST: AVOID using 'await' here unless absolutely necessary". The migration loop awaits two `cfg.update` calls *per workspace folder*, then awaits `reloadSettings` (sync, free) — for N workspaces this is up to 2N round-trips before any subsequent activation step (status bar init, discoveryUX, parser kickoff) runs. On a 4-root multi-root workspace with slow disk this is measurably bad; on a workspace where `cfg.update` rejects (read-only) each rejection is caught by `migrateScopedSetting` but still cost a round-trip. The migrations are independent across workspaces — they should run concurrently.
**Fix:**
```ts
const results = await Promise.all(
  getUrisOfWkspFoldersWithFeatures().map(async wkspUri => {
    let migrated = false;
    try {
      migrated = await migrateLegacyFeaturesPath(wkspUri);
      await migrateLegacySuppressMultiConfig(wkspUri);
      config.reloadSettings(wkspUri);
    } catch (e) {
      config.logger.logInfo(`Phase 15/16 migration error: ${e}`, wkspUri);
    }
    return { wkspUri, migrated };
  }),
);
const pendingFeaturesPathNotifs = results.filter(r => r.migrated).map(r => r.wkspUri);
```
Same correctness, fan-out parallelism.

## WARNING findings

### W-01 — `migrateScopedSetting` `kind: 'write'` always removes source even when destination value is unchanged
**File:** `src/notifications.ts:186-191, 252-258`
**Issue:** In both `migrateLegacySuppressMultiConfig` and `migrateLegacyFeaturesPath`, when the destination already contains the entry being migrated, the transform returns `{ kind: 'write', value: current }` — i.e., write the *unchanged* destination, then remove source. This issues an unnecessary `cfg.update` that triggers a `onDidChangeConfiguration` event, which triggers `configurationChangedHandler`, which calls `clearScanResultCache` only if scan-affecting keys changed (it isn't, here) but still calls `reloadSettings` for every workspace, re-creates watchers, and re-fires `clearTestItemsAndParseFilesForAllWorkspaces`. On idempotent re-activation (extension reload after the migration already ran once), this is a wasted full reparse cycle.
**Fix:** Add a third `TransformResult` variant `{ kind: 'removeOnly' }` or detect dest-equality in the primitive and skip the dest write. The cleanest expression:
```ts
if (destAtScope !== undefined && deepEqual(destAtScope, result.value)) {
  // dest unchanged — only remove source
  await sourceCfg.update(opts.sourceKey, undefined, target);
  return true;
}
```

### W-02 — `migrateLegacyFeaturesPath` only checks the most-specific scope per namespace; cross-scope legacy values silently lost
**File:** `src/notifications.ts:111-122`
**Issue:** The primitive's "most-specific-wins" detection picks ONE scope (workspaceFolder → workspace → global) per `migrateScopedSetting` invocation. If a user has `featuresPath` set at *both* workspaceFolder and global scope (a real scenario for users who set a global default and overrode per-folder), only the workspaceFolder one migrates; the global one is never touched and lingers as a deprecated key forever. The Phase 16 wrapper also calls the primitive once per *namespace* (gs-behave-bdd, behave-vsc), but never per-scope.
**Fix:** Either (a) document that this is intentional in 16-PATTERNS (cross-scope migration is out of scope) AND surface a warning when other scopes contain a stale value, or (b) loop over all three scopes inside the primitive and migrate each one independently. Option (b) is what the wrapper-level docstring at line 224 ("Cross-scope and cross-namespace independence (D-04)") *claims* the implementation does.

### W-03 — `notifications.ts` swallows update() errors via `logger.logInfo` instead of project-standard error path
**File:** `src/notifications.ts:43-45, 156-160`
**Issue:** The codebase convention (per CLAUDE.md/AI_INSTRUCTIONS.md) is `WkspError` + `logger.showError` for workspace-scoped failures the user should know about, OR `diagLog` for internal-only diagnostics. `logger.logInfo` is the wrong choice for either: an `update()` rejection is not informational (it means a write the user implicitly opted into didn't happen) and it's not visible enough (logged silently to the workspace channel without notification). On a read-only workspace the user clicks "Don't Show Again", nothing visibly happens, and they'll click it again next session.
**Fix:** Use `config.logger.logSettingsWarning(...)` for `suppressNotification` failures (per the existing pattern in `Logger`) so the user sees a one-shot notification "Could not persist 'Don't Show Again'", and `diagLog` for the internal migration paths where surfacing to the user adds no value.

### W-04 — `showSuppressibleNotification` returns `undefined` for both "user dismissed" AND "user clicked DSA"; callers can't distinguish
**File:** `src/notifications.ts:55-72`
**Issue:** The comment at line 53-54 says DSA is "intercepted internally — never returned". Fine — but it returns `undefined` on dismiss, on suppress, AND on DSA. The caller at `extension.ts:177-184` does `.then(action => if (action === '...') ...)` which handles all three identically. That's coincidentally correct *today*, but if you ever add a default-action behavior for the dismiss case (e.g. log "user ignored multi-config notification"), you can't distinguish it from "user explicitly opted out". The dismissal vs. opt-out distinction matters for any future telemetry or UX work.
**Fix:** Either return a discriminated result (`{action: string} | {dismissed: true} | {suppressed: true}`) or document explicitly that the three cases are conflated and add a `// LIMITATION:` comment.

### W-05 — Multi-config notification re-fires on every `updateDiscoveryUX` invocation if not suppressed
**File:** `src/extension.ts:150-185`
**Issue:** `updateDiscoveryUX` is called from at least three places: activation (line 356), `configurationChangedHandler` (line 1061), and `startWatchingConfigFiles` (passed as a callback — fires on config-file changes). Unlike the malformed-config notification at line 129, which dedupes via `notifiedConfigErrors`, the multi-config notification has *no* per-session dedup — it relies entirely on `suppressedNotifications`. So a user who has NOT clicked "Don't Show Again" gets a fresh notification every time `configurationChangedHandler` fires (e.g. they edit any unrelated `gs-behave-bdd` setting). This is a regression from Phase 15's intent: the suppression mechanism was supposed to give users control, not to be the *only* gate.
**Fix:** Add a session-scoped `Set<string>` (analogous to `notifiedConfigErrors`) keyed on `${wkspUri.toString()}:multiConfigNotification` that prevents re-firing within the same activation session, regardless of suppression state.

### W-06 — `configuration.ts` `workspaceSettings` getter `try/catch` swallows WkspError silently in production
**File:** `src/configuration.ts:80-94` (commit 26948cc)
**Issue:** Per the commit message this was added so "a single misconfigured workspace doesn't poison iteration / settings access for unrelated workspaces". That's defensible — but the catch-and-`diagLog` path means a user with a genuinely broken `featuresPaths` config (e.g. typo in a path) sees NOTHING in the UI on extension reload. The original code path was: WkspError → activate's per-workspace try/catch (extension.ts:243) → `config.logger.showError`. The activate-time path *also* calls the getter (transitively, via `reloadSettings` etc.), but now that the getter swallows, the user sees the error only on the FIRST settings-loading code path that calls `reloadSettings(wkspUri)` directly (which throws unswallowed per the comment). For multi-workspace setups where workspace B's bad config is encountered first via the iterating getter (e.g. integration test reading workspace A), the error is now demoted to a `diagLog` line that's invisible without xRay.
**Fix:** Keep the swallow but surface ONCE via the workspace's logger so the user gets a one-shot "Behave BDD failed to load settings for workspace X — see output" notification. Track per-uri which ones have been surfaced to avoid spam:
```ts
} catch (e) {
  if (!this._failedSettingsWorkspaces.has(wkspUri.path)) {
    this._failedSettingsWorkspaces.add(wkspUri.path);
    this.logger.showError(e, wkspUri);
  }
  diagLog(`workspaceSettings getter: skipping ${wkspUri.path} due to: ${e}`, wkspUri);
}
```

### W-07 — `normalizePathEntry` is duplicated in `notifications.ts` and `settings.ts`
**File:** `src/notifications.ts:206-208`, `src/settings.ts:195, 215`
**Issue:** The comment at notifications.ts:202-204 explicitly calls out: "The regex MUST be byte-identical to src/settings.ts:204 / L214. If they ever drift, dedup silently double-appends (Pitfall 9 from 16-PATTERNS.md)." This is a known landmine — and the mitigation is a comment, not code. The fix is trivial: export `normalizePathEntry` from `common.ts` and import in both places. The 16-03 SUMMARY presumably had reasons not to (cyclic imports?), but `common.ts` already exports many simple string utils with no cycles back to settings/notifications.
**Fix:** Move `normalizePathEntry` (or its regex `/^\\|^\/|\\$|\/$/g`) to `common.ts` as `normalizeFeaturesPathEntry`, import in both. Removes the "must stay in sync" landmine.

### W-08 — `testWorkspaceConfig.ts` mock has no test coverage for `featuresPath` (singular) → ladder skip
**File:** `src/testWorkspaceConfig.ts:16, 38, 54, 85-86, 140-141`
**Issue:** Phase 16-06 dropped the singular `featuresPath` field from the mock entirely. That's the right call for the *new* code path (settings.ts no longer reads it), but the existing migration logic in `notifications.ts` reads `featuresPath` via real `vscode.workspace.getConfiguration().inspect()` — which the integration tests can't stub via `TestWorkspaceConfig` (the mock surface no longer has a key). So the migration's behavior *under integration test* is unverifiable: any integration test that wants to assert "starting from a workspace with legacy `featuresPath: 'tests/features'`, after activate the value lands in `featuresPaths`" cannot construct the precondition. Unit tests with sinon-stubbed `vscode.workspace.getConfiguration` can do this; but the only way an integration test can exercise the migration is to commit a `.vscode/settings.json` with the legacy key into the example-projects fixture — which the audit doc does not show happening.
**Fix:** Either add a fixture under `example-projects/` with a legacy `featuresPath` setting AND assert post-activate that it has been migrated, OR document explicitly in 16-VERIFICATION which test layer covers this and acknowledge that the mock simplification reduced integration-test coverage of the migration.

## NIT findings

### N-01 — Misleading `Pitfall 8` comment on `reloadSettings`
**File:** `src/extension.ts:328`
**Issue:** The comment "Pitfall 8 — sync, no await" reads as if there were a pitfall in awaiting it. Awaiting a sync void-returning function is a no-op (you `await undefined`), it's just stylistically odd. The real reason is "it's sync"; phrasing it as a pitfall implies something subtler.
**Fix:** Replace with `// reloadSettings is synchronous — no await needed`.

### N-02 — `FEATURES_PATH_NAMESPACES` const is `as const` but typed only via inference
**File:** `src/notifications.ts:199`
**Issue:** Cosmetic. `["gs-behave-bdd", "behave-vsc"] as const` gives a tuple of literals, but the `for (const sourceNs of ...)` iteration treats them as plain string. Could be typed as `readonly ("gs-behave-bdd" | "behave-vsc")[]` for clarity if a future grep wants to find all namespace literals.
**Fix:** Optional. Leave as-is.

### N-03 — `nonUserSettableWkspSettings` in `settings.ts:332-334` lists obsolete keys
**File:** `src/settings.ts:332-334`
**Issue:** The list includes `"featuresUri"`, `"workspaceRelativeFeaturesPath"`, `"projectRelativeFeaturesPath"` (singular) but not the corresponding plural forms — but actually it DOES list `featuresUris`, `stepsSearchUris`, `projectRelativeFeaturesPaths`, `workspaceRelativeFeaturesPaths`. That's correct. However the line 340 filter expression also mentions `workspaceRelativeProjectPath` and `projectRelativeFeaturesPath` (singular) as standalone strings — those are getter properties whose names will *also* surface in `Object.entries(this)` only if they're enumerable own properties (they are getters so they are NOT enumerable), so the filter is dead code for those keys. Harmless but confusing.
**Fix:** Optional cleanup — drop the singular getter names from line 340 filter.

### N-04 — `discoveryDepth` re-read on every `hasFeaturesFolder` call instead of cached
**File:** `src/common.ts:351`
**Issue:** Inside the `<1ms` hot path, `vscode.workspace.getConfiguration("gs-behave-bdd", folder.uri).get<number>("discoveryDepth")` is called once per workspace folder per cache miss. `vscode.workspace.getConfiguration` is not free — it walks the scope chain. For a single-root workspace this is one call so it's fine; for 10-root workspaces with frequent cache invalidation this could matter. Performance is out of scope per the review brief, flagging only because the function carries an explicit <1ms contract.
**Fix:** Out of scope; flagging for awareness.

### N-05 — `extension.ts` unused-comment imports `configurationChangedHandler` twice in flow comments
**File:** `src/extension.ts:962-1069`
**Issue:** Cosmetic. The handler is defined inline as an arrow `const configurationChangedHandler = ...` after being *referenced* at line 678 inside `selectProjectCommand`'s `vscode.window.withProgress` callback. This works because of TS's hoisting of `const` declarations within the same closure, but it's a temporal-dead-zone foot-gun — if anyone refactors the activation flow and that callback fires earlier than expected, you get a ReferenceError. Worth a `// forward-referenced — defined below` comment, or hoist the `const` definition above the command registration block.
**Fix:** Add a comment, or restructure activate() so handler defs appear before subscribers.
